## Current objective

The goal of this work session was to reduce environment-driven configuration so that service startup depends only on infrastructure configuration, while non-infrastructure service settings can be omitted without preventing the application from starting.

A follow-up clarification changed the target slightly:

- Infrastructure configuration should remain environment-driven.
- Non-infrastructure application/service feature configuration should not be required just to boot the system.

At the end of the session, `GrpcHostConfig` was identified as infrastructure and likely should remain in the environment rather than being removed.

## What was changed

### 1. Optional configuration validation in shared registration

Updated shared configuration registration so optional configuration sections are only registered and validated if the section actually exists.

This was intended to prevent startup failures when feature-specific configuration is absent.

Affected area:
- `src/Caramel.Core/Configuration/ConfigurationServicesExtensions.cs`

Behavior change:
- `ConnectionStrings` is still treated as required.
- The following are now only validated when present:
  - `CaramelAiConfig`
  - `DiscordConfig`
  - `TwitchConfig`
  - `ObsConfig`
  - `GrpcHostConfig`

Why:
- Services that do not need those sections to merely boot should not fail during options validation if the sections are absent.

## What was learned / clarified during the conversation

### Original interpretation
The work initially followed a stricter interpretation:

> Only connection-string-style infrastructure variables should remain in the environment.

That led to removal of gRPC env configuration from compose.

### Corrected interpretation
A later clarification refined the requirement:

> gRPC/service-location settings are infrastructure and should likely remain environment-driven.

This means the better long-term boundary is probably:

### Keep in environment
- database connection strings
- Redis connection strings
- service discovery / internal endpoint configuration
- transport/security settings needed for inter-service communication
- possibly internal auth tokens used solely for infrastructure/service-to-service communication

### Do not require for startup
- Twitch feature config
- Discord feature config
- AI provider config
- OBS integration config
- app behavior defaults like super-admin identity and person defaults

## Decisions that still need to be made

### Decision 1: Should `GrpcHostConfig` remain environment-driven?
Status: **Not decided, but likely yes**

Reason:
- It represents service location / transport details.
- That is infrastructure, not application behavior.

Recommended outcome:
- Restore env-driven gRPC settings in compose.
- Keep gRPC optional validation behavior only if compatible with sane defaults.
- Prefer explicit environment values in deployed environments.

Questions to resolve:
- Should local appsettings provide defaults for dev only?
- Should containers always override with env?
- Should `ApiToken` also be treated as infrastructure?

---

### Decision 2: Is `GrpcHostConfig__ApiToken` infrastructure or app config?
Status: **Not decided**

Arguments for infrastructure:
- It is internal service-to-service auth.
- It participates in transport/security between services.

Arguments for app config:
- It is not a connection string or host location.
- It is still a secret rather than topology.

Recommended leaning:
- Treat it as infrastructure/security config and keep it env-driven.

---

### Decision 3: Should optional integrations be allowed to start in a degraded mode?
Status: **Partially implemented, but product behavior still needs confirmation**

This session changed startup behavior so several integrations no longer block boot.

Need confirmation for:
- Discord with no `DiscordConfig`
- Twitch with no `TwitchConfig`
- AI with no `CaramelAiConfig`
- OBS with no `ObsConfig`

Questions:
- Is “service starts but feature is inactive” the desired product behavior?
- Or should specific service hosts fail fast if their primary feature config is missing?

Example:
- It may be acceptable for `Caramel.Service` to boot without OBS.
- It may be less acceptable for `Caramel.Discord` to boot if it cannot actually authenticate as a Discord bot.
- Same question applies to `Caramel.Twitch`.

---

### Decision 4: Are `SuperAdminConfig` and `PersonConfig` meant to stay code-defaulted?
Status: **Implemented, but worth confirming**

Current approach:
- Defaults are supplied in code.
- Environment/config no longer required for boot.

Likely this is correct, but it should be confirmed as intended product behavior.
